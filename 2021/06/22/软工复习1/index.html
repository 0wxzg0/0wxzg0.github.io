1 overall

outline

  • Software Development and Systems Analysis and Design 软件开发和系统分析与设计

  • Introduction to Ridgeline Mountain Outfitters RMO相关介绍

learning objectives

After reading this chapter, you should be able to:

  • Describe the purpose of systems analysis and design in the development of information systems 描述系统分析和设计在信息系统开发中的目的
  • Describe the characteristics of iterative systems development 描述迭代系统开发的特点
  • Explain the six core processes of the Systems Development Life Cycle 解释系统开发生命周期的六个核心流程
  • Identify key documents that are used in planning a project 确定用于规划项目的关键文件
  • Identify key diagrams used in systems analysis and systems design 识别系统分析和系统设计中使用的关键图表
  • Explain the utility of identifying use cases in systems development 解释在系统开发中识别用例的效用
  • Explain the utility of identifying object classes in systems development 解释在系统开发中识别对象类的效用

overview

  • This text is about developing information systems that solve an organization need. 本书是关于开发解决组织需求的信息系统。
  • Chapter 1 takes you through the process of developing one rather small information system 第1章将带您了解开发一个相当小的信息系统的过程
  • The rest of the text elaborates on the basic processes shown in chapter 1 正文的其余部分详细说明了第1章中显示的基本过程
  • Computer application (app) – a computer software program that executes on a computing device to carry out a specific set of functions 计算机应用程序：在计算设备上执行以执行一组特定功能的计算机软件程序
    • Modest scope
  • Information system – a set of interrelated components that collects, processes, stores, and provides as output the information needed to complete business tasks 信息系统：一组相互关联的组件，用于收集、处理、存储并提供完成业务任务所需的信息作为输出
    • Broader in scope than “app”
    • Includes database and related manual processes 包括数据库和相关的手动流程
  • Project – a planned undertaking that has a beginning and end and that produces some definite result 项目：有开始和结束并产生一定结果的计划性事业
    • Used to develop an information system
    • Requires knowledge of systems analysis and systems design tools and techniques
  • Systems analysis – those activities that enable a person to understand and specify what an information system should accomplish 系统分析：使人们能够理解和指定信息系统应该完成什么的活动
  • Systems design – those activities that enable a person to define and describe in detail the system that solves the need 系统设计：使人们能够详细定义和描述解决需求的系统的那些活动
  • System development lifecycle (SDLC) – the entire process consisting of all activities required to build, launch, and maintain an information system 系统开发生命周期：由构建、启动和维护信息系统所需的所有活动组成的整个过程
    • Identify the problem or need and obtain approval 确定问题或需求并获得批准
    • Plan and monitor the project
    • Discover and understand the details of the problem or need
    • Design the system components that solve the problem or satisfy the need
    • Build, test, and integrate system components 构建、测试和集成系统组件
    • Complete system tests and then deploy the solution 完成系统测试，然后部署解决方案
  • Information systems development process – the actual approach used to develop a particular information system (aka: methodology) 信息系统开发过程：用于开发特定信息系统的实际方法（又名：方法论）
    • Unified process (UP) 统一流程
    • Extreme programming (XP) 极限编程
    • Scrum 敏捷开发过程
  • Most processes/methodologies now use Agile and Iterative development 大多数流程/方法现在使用敏捷和迭代开发
  • Agile development – an information system development process that emphasizes flexibility to anticipate new requirements during development 敏捷开发：一种信息系统开发过程，强调在开发过程中预测新需求的灵活性
    • Fast on feet; responsive to change
  • Iterative development – an approach to system development in which the system is “grown” piece by piece through multiple iterations 迭代开发：一种系统开发方法，其中系统通过多次迭代“成长”
    • Complete small part of system (mini-project), then repeat processes to refine and add more, then repeat to refine and add more, until done

summary

  • This text is about developing information systems that solve an organization need 关于开发解决组织需求的信息系统
  • Chapter 1 takes you through the whole process for one small information system
  • System development involves 6 core processes, known as the SDLC 系统开发涉及6个核心流程，称为SDLC
  • The rest of the text elaborates on the basic processes shown in chapter 1

learning objectives

  • Compare the underlying assumptions and uses of a predictive and an adaptive system development life cycle (SDLC) 比较预测性和适应性SDLC的基本假设和使用
  • Describe the key activities and tasks of information system support 描述信息系统支持的关键活动和任务
  • Explain what comprises a system development methodology—the SDLC as well as models, tools, and techniques 解释系统开发方法的组成部分——SDLC 以及模型、工具和技术
  • Describe the two overall approaches used for software construction and modeling: the structured approach and the object-oriented approach 描述用于软件构建和建模的两种总体方法：结构化方法和面向对象方法
  • Describe the key features of Agile development 描述敏捷开发的主要特征

overview

  • Later chapters focused on Systems Analysis activities and tasks and some System Design activities and tasks 后面的章节侧重于系统分析活动和任务以及一些系统设计活动和任务
  • Now we return to look at the SDLC and related concepts in more detail
    • Predictive versus Adaptive SDLC variations 预测与自适应 SDLC 变化
    • Activities and Tasks of System Support 系统支持的活动和任务
    • Models, Methodologies, Tools and Techniques 模型、方法、工具和技术
    • Impacts of Traditional versus OO development 传统与面向对象开发的影响
    • Agile Development 敏捷开发

summary

  • This chapter covers approaches to system development in more detail
  • There are two approaches to the SDLC: Predictive and Adaptive
  • A predictive SDLC, also known as the waterfall model, is used when it is possible to plan the project completely in advance 预测性SDLC，也称为瀑布模型，在可以提前完全规划项目时使用
  • An Adaptive SDLC, which uses iteration, is used when the requirements are less certain and the project will need to react to changes 当需求不太确定并且项目需要对变化做出反应时，使用迭代的自适应SDLC
    • This text uses an adaptive approach to the SDLC
  • All new information systems require support once completed 所有新的信息系统完成后都需要支持
  • System development project use a methodology (or development process) and many are available. A methodology includes an SDLC and tools, techniques, and models 系统开发项目使用一种方法论（或开发过程），并且很多都是可用的。方法论包括 SDLC 和工具、技术和模型。
  • There are two approaches to construction and modeling software: the traditional structured approach and the newer object-oriented approach 构建和建模软件有两种方法：传统的结构化方法和较新的面向对象方法
  • Agile development is the current trend in system development 敏捷开发是当前系统开发的趋势

3 Project Management

outline

  • Principles of Project Management 项目管理原则
  • Activities of SDLC Core Process 1:
    • Identify the Problem and Obtain Approval 找出问题并获得批准
  • Activities of SDLC Core Process 2:
    • Plan and Monitor the Project 计划和监控项目

learning objectives

  • Describe the factors that cause a software development project to succeed or fail 描述导致软件开发项目成功或失败的因素
  • Describe the responsibilities of a project manager 描述项目经理的职责
  • Describe the knowledge areas in the project management body of knowledge (PMBOK) 描述项目管理知识体系(PMBOK)中的知识领域
  • Describe the Agile approach to the project management knowledge areas 描述项目管理知识领域的敏捷方法
  • Explain the activities required to get a project approved (Core Process 1) 解释获得项目批准所需的活动（核心流程1）
  • Explain the activities required to plan and monitor a project (Core Process 2) 解释计划和监控项目所需的活动（核心流程2）

overview

  • Chapter 8 covered the various alternatives for the SDLC and approaches to development 涵盖了SDLC的各种替代方案和开发方法
  • You should be asking yourself:
    • “How are all these activities coordinated?”
    • “How do I know which tasks to do first?”
    • “How is the work assigned to the different teams and team members?”
    • “How do I know which parts of the new system should be developed first?”

summary

  • Project management is an important and challenging career role. Information systems projects do not have a great success rate, and project management knowledge and skills are valued and needed to improve this record. 项目管理是一个重要且具有挑战性的职业角色。信息系统项目的成功率并不高，项目管理知识和技能很有价值，需要改善这一记录。
  • Project management is directing other people to achieve a planned result on schedule and on budget. Project managers have internal and external responsibilities. 项目管理正在指导其他人按计划和预算实现计划的结果。项目经理有内部和外部的责任。
  • Project managers work with clients, who fund the project, an oversight committee which approves and reviews progress, and users who will directly interact with the system. 项目经理与为项目提供资金的客户、批准和审查进度的监督委员会以及将直接与系统交互的用户一起工作。
  • The discipline of project management is organized into the Project Management Body of Knowledge (PMBOK) that includes nine knowledge areas. Project managers should study and digest this body of knowledge. 项目管理学科被组织成项目管理知识体系 (PMBOK)，其中包括九个知识领域。项目经理应该学习和消化这些知识体系。
  • Managing a project can be at a high or low level of ceremony, meaning the degree that the project management processes are formal and documented. Agile project management is usually used with adaptive life cycles and low ceremony. 项目管理可以是高级别或低级别的仪式，这意味着项目管理过程正式和记录的程度。敏捷项目管理通常与适应性生命周期和低仪式一起使用。
  • The SDLC in this text includes two Core Processes that involve the project manager: 两个核心流程
    • 1) Identify the problem and obtain approval 找出问题并获得批准
    • 2) Plan and monitor the project 计划和监控项目
  • The core process Identify the problem and obtain approval includes the following activities: 核心流程1（识别问题并获得批准）包括以下活动：
    • 1) identify the problem 找出问题
    • 2) quantify project approval factors 量化项目审批因素
    • 3) perform risk and feasibility analysis 进行风险和可行性分析
    • 4) review with client and obtain approval 与客户一起审查并获得批准
  • A key deliverable is the System Vision Document, which includes a problem description, an overview of system capabilities, and a list of business benefits. Key project approval factors include time estimate, cost estimate, and cost/benefit analysis. Additionally, risk and feasibility factors are organizational, technological, resource, and schedule. 一个关键的可交付成果是系统愿景文档，其中包括问题描述、系统功能概述和业务收益列表。关键的项目批准因素包括时间估算、成本估算和成本/收益分析。此外，风险和可行性因素是组织、技术、资源和进度。
  • The core process Plan and monitor the project includes the following activities: 核心过程2（计划和监控项目）包括以下活动：
    • 1) establish the project environment 建立项目环境
    • 2) schedule the work 安排工作
    • 3) staff and allocate resources 人员和资源分配
    • 4) evaluate work processes 评估工作流程
    • 5) monitor progress and make corrections 监控进度并进行更正
  • Scheduling the work includes a project iteration schedule and detailed work schedules. A work breakdown structure (WBS) lists tasks to be completed. Dependencies and time estimates are also considered and shown in a Gantt chart. 安排工作包括项目迭代计划和详细的工作计划。工作分解结构(WBS)列出了要完成的任务。依赖关系和时间估计也被考虑并显示在甘特图中。

outline

learning objectives

  • Describe the activities of systems analysis 描述系统分析的活动
  • Explain the difference between functional and nonfunctional requirements 解释功能性需求和非功能性需求之间的区别
  • Describe the role of models in systems analysis 描述模型在系统分析中的作用
  • Identify and understand different kinds of stakeholders and their contributions to requirements definition 识别和理解不同类型的利益相关者及其对需求定义的贡献
  • Describe information-gathering techniques and determine when each is best applied 描述信息收集技术并确定最佳应用时间

overview

  • This chapter expands the SDLC processes to cover a wider range of concepts, tools and techniques 本章扩展了SDLC流程以涵盖更广泛的概念、工具和技术
  • Core process 3: Discover and understand the details of the problem or need—is the main focus of systems analysis 核心流程3：发现并理解问题或需求的细节是系统分析的主要焦点
  • Systems analysis activities are detailed in this chapter 本章详细介绍了系统分析活动
  • A larger Ridgeline Mountain Outfitters (RMO) project is introduced that will be used throughout the text to illustrate analysis and design 介绍了一个更大的(RMO) 项目，该项目将贯穿全书以说明分析和设计

summary

  • Systems analysis activates correspond to the core SDLC process Discover and understand details 系统分析激活了对应核心SDLC流程“发现并了解细节”
  • System projects originate from the information system strategic plan, which contains an technology architecture plan and an application architecture plan 系统项目源于信息系统战略规划，包括技术架构规划和应用架构规划
  • The RMO CSMS Project will be used throughout the text as an example of analysis and design “RMO CSMS” 项目将贯穿全书作为分析和设计的示例
  • Systems analysis involves defining system requirements– functional and non-functional 系统分析涉及定义系统需求——功能性和非功能性
  • Analysis activities include 分析活动
    • Gather detailed information 收集详细信息
    • Define requirements 定义需求
    • Prioritize requirements 优先考虑需求
    • Develop user-interface dialogs 开发用户界面对话框
    • Evaluate requirements with users 与用户一起评估需求
  • FURPS+ is the acronym for functional, usability, reliability, performance, and security requirements FURPS+是功能、可用性、可靠性、性能和安全要求的首字母缩写词
  • Models and modeling are used to explore and document requirements 模型和建模用于探索和记录需求
  • A model represents some aspect of a system, and can include textual, graphical, and mathematical models 模型代表系统的某些方面，可以包括文本、图形和数学模型
  • Unified Modeling Language (UML) is the standard set of notations and terminology for information systems models 统一建模语言 (UML) 是信息系统模型的标准符号和术语集
  • Stakeholders are the people who have an interest in the success of the project 利益相关者是对项目的成功感兴趣的人
  • There are internal vs. external stakeholders and operational vs. executive stakeholders 存在内部与外部利益相关者、运营与执行利益相关者
  • Information gathering techniques are used to collect information about the project 信息收集技术用于收集有关项目的信息
    • Interviews, questionnaires, reviewing documents, observing business processes, researching vendors, comments and suggestions 访谈、问卷调查、审阅文件、观察业务流程、研究供应商、意见和建议
  • The UML Activity Diagram is used to document (model) workflows after collecting information UML活动图用于在收集信息后记录（建模）工作流

5 Use Case Diagram

outline

  • Use Cases and User Goals
  • Use Cases and Event Decomposition
  • Use Cases and CRUD
  • Use Cases in the Ridgeline Mountain Outfitters Case
  • Use Case Diagrams

learning objectives

  • Explain why identifying use cases is the key to defining functional requirements 解释为什么识别用例是定义功能需求的关键
  • Describe the two techniques for identifying use cases 描述识别用例的两种技术
  • Apply the user goal technique to identify use cases 应用用户目标技术来识别用例
  • Apply the event decomposition technique to identify use cases 应用事件分解技术来识别用例
  • Apply the CRUD technique to validate and refine the list of use cases 应用CRUD技术来验证和改进用例列表
  • Describe the notation and purpose for the use case diagram 描述用例图的符号和使用目的
  • Draw use case diagrams by actor and by subsystem 按参与者和子系统绘制用例图

overview

  • This chapter focuses on identifying and modeling the key aspect of functional requirements– use cases 本章重点介绍功能需求的关键方面的识别和建模：用例
  • In the RMO Tradeshow System from Chapter 1, some use cases are Look up supplier, Enter/update product information, Enter/Update contact information 在第1章的RMO Tradeshow System中，一些用例是查找供应商、输入/更新产品信息、输入/更新联系信息
  • In this chapter’s opening case Waiters on Call, examples of use cases are Record an order, Record delivery, Update an order, Sign in driver, Reconcile driver receipts, Produce end of day deposit slip, and Produce weekly sales reports 在本章的开篇案例Waiters on Call中，用例的示例包括记录订单、记录交付、更新订单、登录驱动程序、核对驱动程序收据、生成日终存款单和生成每周销售报告

summary

  • This chapter is the first of three that focuses on modeling functional requirements as a part of systems analysis 本章重点是将功能需求建模作为系统分析的一部分
  • Use cases are the functions identified, the activities the system carries out usually in response to a user request 用例是识别的功能，系统通常响应用户请求而执行的活动
  • Two techniques for identifying use cases are the user goal technique and the event decomposition technique 识别用例的两种技术是用户目标技术和事件分解技术
  • The user goal technique begins by identifying end users called actors and asking what specific goals they have when interacting with the system 用户目标技术首先识别称为参与者的最终用户，并询问他们在与系统交互时有什么具体目标
  • The event decomposition technique begins by identifying events that occur that require the system to respond. 事件分解技术首先识别发生的需要系统响应的事件。
  • Three types of events include external, temporal, and state events 三种类型的事件包括外部事件、时间事件和状态事件
  • Brief use case descriptions are written for use cases 为用例编写简短的用例描述
  • The CRUD technique is used to validate and refine the use cases identified CRUD技术用于验证和改进已识别的用例
  • The use case diagram is the UML diagram used to show the use cases and the actors 用例图是用于显示用例和参与者的UML图
  • The use case diagram shows the actors, the automation boundary, the uses cases that involve each actor, and the <<includes>> relationship. 用例图显示了参与者、自动化边界、涉及每个参与者的用例以及包含关系。
  • A variety of use case diagrams are draw depending on the presentation needs of the analysis 根据分析的呈现需求绘制各种用例图

6 Use Case Descriptions

outline

  • Use Case Descriptions
  • Activity Diagrams for Use Cases

learning objectives

  • Write fully developed use case descriptions 编写完全开发的用例描述
  • Develop activity diagrams to model flow of activities 创建活动图以模拟活动流
  • Develop system sequence diagrams 创建系统时序图
  • Develop state machine diagrams to model object behavior 建立状态机图以对对象行为建模
  • Explain how use case descriptions and UML diagrams work together to define functional requirements 解释用例描述和UML图如何协同工作来定义功能需求

overview & summary

  • This chapter focuses on additional techniques and models to extend the requirements models to show more detail 本章重点介绍扩展需求模型以显示更多细节的其他技术和模型
  • Fully developed use case descriptions provide information about each use case, including actors, stakeholders, preconditions, post conditions, the flow of activities and exceptions conditions 完全开发的用例描述提供有关每个用例的信息，包括参与者、利益相关者、前提条件、后置条件、活动流程和异常条件
  • Activity diagrams (first shown in Chapter 2) can also be used to show the flow of activities for a use case 活动图也可用于显示用例的活动流（第2章中首次显示）
  • System sequence diagrams (SSDs) show the inputs and outputs for each use case as messages 系统序列图(SSD)将每个用例的输入和输出显示为消息
  • State machine diagrams show the states an object can be in over time between use cases 状态机图显示了一个对象在用例之间随着时间的推移可以处于的状态
  • Use cases are modeled in more detail using fully developed use case descriptions, activity diagrams, and system sequence diagrams 使用完全开发的用例描述、活动图和系统序列图对用例进行更详细的建模

  • Not all use cases and domain classes are modeled at this level of detail. Only model when there is complexity and a need to communicate details 并非所有用例和领域类都在此详细级别建模。只有在存在复杂性和需要沟通细节时才建模。

11 几种设计模式

outline

  • Factory Pattern 工厂模式
    • Simple Factory 简单工厂模式
    • Factory Method 工厂方法模式
    • Abstract Factory 抽象工厂模式
  • Singleton Pattern 单例模式
  • Adapter Pattern 适配器模式
  • Proxy Pattern 代理模式

summary

工厂模式：抽象了对象的构造过程，即生成对象的过程。使得具有类似功能的对象可以具有一个共同的或者统一的生成方法。

  • 简单工厂：

    • 用来生产同一类型的已知多种固定产品
    • 不能增加新产品（否则要修改原有代码）
    • 不同产品通过同一个工厂类的同一个成员函数生成
    • 使用静态方法生成产品对象，通过输入参数的不同来返回不同对象实例

  • 工厂方法：

    • 用来生产同一类型中已知多种固定产品
    • 可以增加该类型的新固定产品
    • 不同产品通过不同的工厂类生成，一个类中用一个成员函数生产

  • 抽象工厂：

    • 用来生产不同产品族的全部固定产品
    • 可以增加新产品族（不同产品的组合），增加类来增加新产品
    • 不同产品族通过不同的工厂类生成，一个类中有多个成员函数去生成不同具体产品

单例模式：

  • 作用：
    • 保证一个类仅有一个实例，并提供一个访问它的全局访问点。
    • 代替全局对象。
  • 特点：
    • 通过静态函数来生成唯一实例。
    • 构造函数是私有的。

适配器模式：

  • 作用：
    • 将一个类的接口，转换成客户期望的另一个接口。
    • 让原本接口不兼容的类可以协同工作（无间合作）。
  • 特点：
    • 提高了现有类的重用性，解决了现有类和复用环境要不一致的问题
    • 低耦合，无需修改客户端原有代码

代理模式：

  • 作用：
    • 增加一层，从自身以外的接口向这个对象转发请求，为另一个对象提供间接性访问（此处与适配器模式相同）
    • 代理提供的接口和原本的接口是一样的
    • 代理模式的作用是不把实现直接暴露给client，而是通过代理这个层，代理能够做一些处理
  • 特点：
    • 职责清晰：实现实际的业务逻辑，不用关心其他非本职责的事务，通过后期的代理完成一件完成事务，附带的结果就是编程简洁清晰。
    • 降低耦合：起到了中介的作用和保护了目标对象的作用。
    • 高扩展性

  • Implementation activities occur before system is turned over to users 实施活动发生在系统移交给用户之前

summary

  • Implementation activities occur after design and before system is turned over to users 实施活动发生在设计之后和系统移交给用户之前
  • Implementation is complex
    • Interdependence of programming, quality assurance, hardware and software installation, documentation, and training 编程、质量保证、硬件和软件安装、文档和培训的相互依赖
  • Implementation is difficult to manage
    • Activities must be properly sequenced 活动必须按顺序进行
    • Progress must be continually monitored 必须持续监测进展
  • Implementation is risky
    • Significant time and resources required 需要大量的时间和资源
    • Often affects systems vital to daily operations 通常会影响对日常运营至关重要的系统
  • Software components constructed to 构建软件组件以
    • Minimize development resources needed 最大限度地减少所需的开发资源
    • Maximize ability to test system and control errors 最大限度地提高测试系统和控制错误的能力
    • These goals often conflict: trade-off among resources, time, and desire to correct errors 这些目标经常发生冲突：资源、时间和纠正错误的愿望之间的权衡
  • Data conversion, installation, documentation, and training follow programming and testing 数据转换、安装、文档和培训遵循编程和测试
  • Installed and documented system is prerequisite for complete training 安装和记录系统是完成培训的先决条件
  • Fully populated database needed to begin operation 开始操作需要完全填充的数据库
  • Support activities occur after system becomes operational and might continue for years to support user requirements and reduce operational risk 支持活动发生在系统投入运行后，可能会持续数年以支持用户需求并降低运营风险
